# Rollback and Revert Planning

## 1. Domain Theory and Conceptual Foundations
Rollback and revert planning is a core operational safety constraint in modern software configuration management and deployment engineering. As outlined in the IEEE Software Engineering Body of Knowledge (SWEBOK v4) Chapter 6 (Software Configuration Management) and Chapter 11 (Software Quality), every deployment, feature release, and infrastructure change must be treated as a potentially disruptive event. To mitigate operational risks, release engineers must design software and data systems to be fully reversible, ensuring that the system can be restored to its last-known stable configuration immediately if an anomaly is detected.

### 1.1 The Principle of Reversibility
Reversibility dictates that for every forward change action (deployment, migration, feature flag enablement), there must exist a matching, fully tested reverse action. Relying on "roll-forward" strategies (writing quick fixes on top of a broken production deployment) is banned for critical path defects, as it prolongs system downtime and increases the risk of compounding failures. The target is to minimize the Mean Time to Recovery (MTTR):
$$MTTR = \frac{\text{Total Downtime}}{\text{Number of Outages}}$$
Fast, automated rollback mechanisms are the primary means of keeping MTTR low.

### 1.2 Database Migration Integrity: Up/Down Scripts and Additive Schemas
Database schema updates present the highest risk during rollbacks, as data once written or modified cannot easily be "undone" without risk of corruption or data loss. SWEBOK guides that database migrations must follow strict structural rules:
- **Up and Down Migrations**: Every schema change script (Up migration) must be paired with an explicit, automated revert script (Down migration). Down migrations must restore the schema structure (tables, columns, indexes, constraints) without corrupting existing data.
- **Additive Migrations**: Schema updates should be additive. Instead of renaming a column or dropping a table in a single deployment, engineers should:
  1. Add the new column (and write to both columns during a transition phase).
  2. Migrate existing data from the old column to the new column.
  3. Update application readers to consume the new column.
  4. Drop the old column only after the release has proved fully stable.

### 1.3 Mixed-Version Concurrency and N-1 Compatibility
During rolling updates, blue-green deployments, or server restarts, there is a transitional window where two distinct versions of the application code run concurrently against the same database schema:
- **Backward Compatibility**: Version $N$ of the application code must be able to read and write data generated by version $N-1$ of the database schema.
- **Forward Compatibility (N-1 Compatibility)**: Version $N-1$ of the application code must be able to read and write data generated by version $N$ of the schema. This ensures that if version $N$ of the application is rolled back to version $N-1$, the older application code will not crash when encountering database records written by the newer, rolled-back version.

### 1.4 Automated Rollback Triggers and Observability
Rollbacks must not depend entirely on manual human decision-making. SWEBOK highlights the importance of integrating observability metrics as automated rollback gates:
- **Error Rate Spikes**: If the HTTP 5xx error rate or unhandled exception count exceeds a predefined threshold (e.g. $> 1\%$) during deployment, the pipeline triggers an automatic rollback.
- **Latency Regressions**: If 95th-percentile response times exceed SLA boundaries.
- **Health Check Failures**: If container probes fail to report a healthy state within a specified startup window.

### 1.5 The Parallel Change (Expand-Contract) Pattern
When performing complex database refactorings that cannot be done in a single backward-compatible step, SWEBOK-aligned operations employ the Parallel Change pattern. This splits the change into three distinct deployment phases:
1. **Expand Phase**: The database schema is expanded to support both the old and new structures concurrently (e.g., adding a new table or column). The application is deployed to write to both the old and new structures, but reads exclusively from the old structure.
2. **Transition Phase**: A background migration job copies existing records from the old structure to the new one. The application is updated to read from the new structure. If a rollback is triggered here, the system is fully functional because the old structure has remained fully updated.
3. **Contract Phase**: Once the new structure has been validated under load, the old database columns or tables are deleted, and the application code that wrote to both structures is cleaned up.

### 1.6 Blue-Green Routing and DNS-Based Rollbacks
In modern deployment architectures, rollbacks are optimized by decoupling code deployment from traffic routing. Under a Blue-Green deployment model:
- **Blue Environment**: The current active production environment running version $N-1$.
- **Green Environment**: The new isolated environment running version $N$.
- **Weight-Based Shifting**: Traffic is progressively shifted (e.g., via a load balancer, reverse proxy, or Cloudflare DNS/Workers) from Blue to Green.
- **Instant Rollback**: If an error trigger is met, routing rules are instantly updated to point 100% of traffic back to the Blue environment. Because the Blue environment remained running and untouched, the rollback is instantaneous, avoiding the latency of re-deploying code packages.

## 2. Standard Operating Procedures (SOP)
The agent must follow these step-by-step procedures to plan and verify rollback procedures:

### Step 2.1: Author Reversible Database Migrations
When database schema modifications are required:
- Create paired migration scripts (e.g., `0001_add_user_profile.up.sql` and `0001_add_user_profile.down.sql`).
- Ensure the `down` script drops the exact columns, tables, or indexes added in the `up` script.
- Test both scripts locally, verifying that migrating up and then immediately migrating down leaves the database in a clean, consistent state.

### Step 2.2: Implement N-1 Application Safeguards
Write application code that tolerates missing or extra columns:
- When query-mapping database results, use optional property mapping and default fallbacks.
- Never write select statements that depend on hardcoded column counts (e.g., avoid `SELECT *` in production, using explicit column selections instead).
- Verify that Zod parsers or validation layers allow extra, unexpected properties without throwing validation errors.

### Step 2.3: Document the Rollback Protocol in the Implementation Plan
Include a **Rollback Plan** section in the `implementation_plan.md` before deploying:
- Document the exact terminal commands required to revert the application deployment (e.g., `wrangler rollback` or container deployment commands).
- Document the command to run database down migrations.
- Identify the team leads or roles responsible for approving manual rollbacks if automated gates are not active.

### Step 2.4: Execute Phased Canary Deployments
Deploy changes to a fraction of the infrastructure (e.g., 5% of traffic):
- Monitor error logs, CPU usage, and latencies in real time.
- If anomalies are detected in the canary nodes, isolate the nodes, route all traffic back to the stable versions, and execute a rollback.

### Step 2.5: Execute the Rollback Procedure
If a deployment fails validation or triggers an alert:
- Run the documented application rollback command to restore the previous code version.
- If the schema must be reverted, run the database down migration scripts.
- Verify that the system has successfully returned to its healthy baseline, and check that no transactions or user sessions were lost.

### Step 2.6: Conduct a Post-Mortem and Root Cause Analysis (RCA)
Once the system is restored to a stable state:
- Perform a 5-Whys analysis to identify why the deployment failed and why the issue was not caught by local regression tests.
- Document the findings in the system logs or walkthroughs, and write the missing tests to prevent future recurrences of the defect.

### Step 2.7: Session and State Reversibility Audits
Verify that user sessions and shared states survive code rollback:
- Audit redis cache schemas and session storage. Ensure that session models in version $N$ do not introduce properties that cause version $N-1$ to crash during deserialization.
- Check that JWT or OAuth tokens issued by version $N$ are fully decryptable and accepted by version $N-1$.
- Confirm that cookie structures do not experience name collisions or incompatible serialization updates.

## 3. Agent Compliance Checklist
The agent must verify compliance with the following criteria regarding rollback and revert planning:

- [ ] **Up/Down Migration Pairing**: Are all SQL schema updates accompanied by matching, verified revert scripts?
- [ ] **Additive Schema Design**: Have column renames or drops been structured as additive changes across multiple releases?
- [ ] **N-1 Compatibility Checked**: Did the agent verify that older application code can read and write to the new schema state?
- [ ] **No SELECT * in Production**: Are all database queries structured with explicit column selections to tolerate schema changes?
- [ ] **Zod Parsing Flexibility**: Do data validation schemas tolerate extra, unknown fields without throwing unhandled exceptions?
- [ ] **Rollback Plan Documented**: Is a step-by-step rollback procedure with exact rollback commands included in `implementation_plan.md`?
- [ ] **MTTR Goals Aligned**: Is the rollback procedure optimized to execute quickly to minimize MTTR?
- [ ] **Automated Alerts Configured**: Are error rates and latency thresholds configured as deployment health monitors?
- [ ] **Canary Staging Planned**: If deploying to production, is there a plan to roll out changes progressively (e.g., canaries)?
- [ ] **No Git Reset Abuse**: Did the agent avoid using global git resets, relying on targeted `git restore` or revert commits?
- [ ] **No Forbidden Terminology**: Has the content been scanned to ensure none of the forbidden workspace words are used?
- [ ] **Size Bounds Validation**: Has the agent verified that the modified rule file remains under the 12,000 character limit?
- [ ] **Escaped Backticks**: Are all backticks inside the rule content escaped to prevent string termination?
- [ ] **Walkthrough Record Completed**: Does `walkthrough.md` document the rollback validation tests performed?
- [ ] **Task List Sync**: Are rollback planning, up/down migration tests, and compatibility reviews tracked in `task.md`?
- [ ] **Index Signature Safety**: Do rollback configurations use bracket notation to access properties on index-signature objects?
- [ ] **Sonar Assertions wrapped**: Do test cases verifying that a void method executes without issues wrap the call in `expect(() => ...).not.toThrow()`?
- [ ] **Explicit Member Access**: Are all methods and properties on migration helper classes declared with explicit accessibility modifiers?
