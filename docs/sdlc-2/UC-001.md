# Guided Architecture & System Design Creation

## 1. Description
This use case covers how a developer uses the assistant's `/sdlc-2` skill to synthesize system architecture and detailed software designs from existing, frozen Phase 1 requirements.

## 2. Actors and Stakeholders
- **Lead Developer / Architect**: The primary actor who interacts with the skill to translate requirements into technical designs.
- **Security Auditor**: Stakeholder who reviews the generated architecture documents for compliance with security policies (OWASP, STRIDE).
- **QA Engineer**: Stakeholder who verifies that the system architecture satisfies all functional requirements and tests are correctly mapped.

## 3. Flow of Events
1. The developer invokes the `/sdlc-2` command at the root of the monorepo.
2. The system locates and validates the SARA requirements files under `docs/sdlc-2/`.
3. The system scans the repository configurations (Wrangler files, GraphQL schemas, database configs) to understand the current topology.
4. The system starts an interactive interview, grilling the developer on:
   - Domain modeling (contexts, value objects, domain events).
   - Database normalization (3NF relational schema).
   - Architecture topology and platform configuration.
   - Security tactics.
5. The developer provides technical details.
6. The system generates SARA-compliant Markdown documents in `docs/sdlc-2/` with embedded Mermaid diagrams, linking back to requirements.
7. The system validates the final combined graph using `sara check`.
