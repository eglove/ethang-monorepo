# GitHub CI Workflow Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Create a GitHub Actions CI workflow that runs lint, test, and build across all monorepo workspaces on every PR to master, blocking merge until all three pass.

**Architecture:** A four-job DAG — `install` primes the pnpm store cache, then `lint`, `test`, and `build` run in parallel. Each parallel job restores the cache and runs `pnpm install --frozen-lockfile` (near-instant on cache hit) before executing its script. The `lint` job appends `git diff --exit-code` to catch ESLint `--fix` auto-modifications that would otherwise silently pass.

**Tech Stack:** GitHub Actions, pnpm 11, Node 24, `pnpm/action-setup@v4`, `actions/setup-node@v4` (built-in pnpm store caching)

### Task 1: Create the CI workflow file

**Files:**
- Create: `.github/workflows/ci.yml`

- [ ] **Step 1: Create the `.github/workflows` directory and write the workflow file**

```yaml
# .github/workflows/ci.yml
name: CI

on:
  pull_request:
    branches: [master]
  push:
    branches: [master]

jobs:
  install:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '24'
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile

  lint:
    needs: install
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '24'
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: git diff --exit-code

  test:
    needs: install
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '24'
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm test

  build:
    needs: install
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '24'
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm build
```

> **Note on pnpm version:** `pnpm/action-setup@v4` reads the `packageManager` field from `package.json` via Corepack. The project currently uses `pnpm@11.0.0-dev.1005+sha512...` — a dev build. If the action fails to resolve this version, pin it explicitly in the action:
> ```yaml
> - uses: pnpm/action-setup@v4
>   with:
>     version: 10  # or latest stable
> ```
> Then update `packageManager` in `package.json` to match.

- [ ] **Step 2: Validate YAML syntax**

Run:
```bash

```
Expected: prints the parsed object with no errors. If `npx` is not available, visually verify indentation — YAML is whitespace-sensitive and incorrect indentation is the most common error.

- [ ] **Step 3: Commit**

```bash

git commit -m "ci: add GitHub Actions workflow for lint, test, and build"
```

### Task 2: Configure branch protection on GitHub

This is a manual step in the GitHub UI — there is nothing to commit.

- [ ] **Step 1: Open branch protection settings**

Navigate to: **GitHub → your repo → Settings → Branches → Add branch protection rule**

- [ ] **Step 2: Configure the rule**

- Branch name pattern: `master`
- Enable: **Require status checks to pass before merging**
  - Search for and add these required checks: `lint`, `test`, `build`
  - (These names come from the `jobs:` keys in `ci.yml`. They appear in the dropdown after at least one workflow run has completed.)
- Enable: **Require branches to be up to date before merging**
- Save the rule

> **Tip:** The status checks only appear in the search dropdown after the workflow has run at least once. Push the `ci.yml` commit to a branch and open a PR first to trigger the run, then come back to add the required checks.
