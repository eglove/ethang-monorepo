# Phase 6 Maintenance and Troubleshooting Automation

## 1. Description
This use case defines how the `/sdlc-6` skill operates as an automated troubleshooting routine for agents. It parses external static reports from the GitHub PR, SonarQube quality gates, and MegaLinter runs, identifies compilation/build, linting, and coverage issues, and applies targeted programmatic fixes in accordance with Phase 6 maintenance rules.

## 2. Actors and Stakeholders
- **AI Troubleshooter Agent**: The primary actor executing `/sdlc-6` to parse failure logs, target files, and apply correctives.
- **Lead Developer / Reviewer**: Evaluates the applied changes on the PR and merges them once CI status is green.
- **CI Environment**: Runs the builds and compiles the SonarQube/MegaLinter reports.

## 3. Preconditions
- The workspace MUST be a valid git repository.
- Modified files exist in the current branch diff compared to the baseline branch.
- CI/CD build logs or reports (GitHub Actions, SonarQube, or MegaLinter) must have run and deposited output files in the designated reports folder.

## 4. Postconditions
- All parsed compiler/typescript, linter, and quality gate issues have been resolved.
- Local validation checks (eslint, vitest, tsc) pass on the affected packages.
- A clean, minimized git diff is produced for the final pull request review.

## 5. Flow of Events
1. The AI Troubleshooter Agent is triggered with `/sdlc-6`.
2. The agent reads the local git diff to identify the modified files and packages in the PR.
3. The agent retrieves the build, SonarQube, and MegaLinter error lists.
4. The agent categorizes each failure according to [maintenance-classification](file:///.agents/rules/maintenance-classification.md).
5. The agent performs a [maintenance-impact-analysis](file:///.agents/rules/maintenance-impact-analysis.md) on the target files to map dependencies.
6. The agent applies targeted code refactoring and fixes (e.g., resolving duplicate strings, typecast loosening, or eslint-disable comments) directly to the files.
7. The agent validates the fixes against only the specific failed files and reporting tools.
8. If the reports are resolved, the workflow transitions to a successful end state. If no reports are found or are inaccessible, the workflow halts and fails immediately.
