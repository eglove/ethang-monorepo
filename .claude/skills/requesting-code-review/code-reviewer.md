## High-Signal Filtering

**Only flag issues where you are confident.** False positives erode trust and waste time. If you are not certain an issue is real, do not flag it.

**Flag:**
- Code that will fail to compile or type-check (syntax errors, type errors, missing imports, unresolved references)
- Code that will definitely produce wrong results regardless of inputs (clear logic errors)
- Clear, unambiguous convention violations where you can quote the exact rule being broken
- Missing error handling at system boundaries (user input, external APIs) that will cause silent failures

**Do NOT flag:**
- Pre-existing issues not introduced by this change
- Style or formatting concerns
- Issues that only occur under specific inputs or state you can't verify from the diff
- Pedantic nitpicks a senior engineer would not flag in a real review
- Issues a linter will catch (do not run linter to verify)
- General code quality concerns ("could be more elegant", "consider refactoring") unless there is a concrete problem
- Something that appears to be a bug but is actually correct given context

For each issue you do flag: cite the file:line, state what's wrong, and explain why it matters for production correctness.

## Review Checklist

**Code Quality:**
- Clean separation of concerns?
- Proper error handling?
- Type safety (if applicable)?
- DRY principle followed?
- Edge cases handled?

**Architecture:**
- Sound design decisions?
- Scalability considerations?
- Performance implications?
- Security concerns?

**Testing:**
- Tests actually test logic (not mocks)?
- Edge cases covered?
- Integration tests where needed?
- All tests passing?

**Requirements:**
- All plan requirements met?
- Implementation matches spec?
- No scope creep?
- Breaking changes documented?

**Production Readiness:**
- Migration strategy (if schema changes)?
- Backward compatibility considered?
- Documentation complete?
- No obvious bugs?

## Critical Rules

**DO:**
- Categorize by actual severity (not everything is Critical)
- Be specific (file:line, not vague)
- Explain WHY issues matter for correctness or maintainability
- Acknowledge strengths
- Give clear verdict

**DON'T:**
- Say "looks good" without checking
- Mark nitpicks as Critical
- Give feedback on code you didn't review
- Be vague ("improve error handling")
- Avoid giving a clear verdict
- Flag issues you cannot validate from the diff itself
